6. Ctrl+Z of Line

About the EOF identifier. As I’ve known before testing, EOF stands for the end of file or error of file, which can indicate that a file is at its end. There’s nothing to say about the case of an actual file. But I’ve noticed that, in my windows, ctrl+z combination can stand for the EOF. However, when running the code below, I got a strange reaction:

When I’ve input Hello, world.^Z, the program did not end after printing Hello, world., Instead it paused at the end of that line waiting for my next input, which means that, in this case(although I temporarily wasn’t sure what kind of case was that case), ctrl+z didn’t mean EOF. After browsing through the internet and get the following output: When ^Z is inside a line input, the characters after the ^Z symbol wasn’t processed(including the ENTER). And when there are characters in front of the ^Z identifier inside a line input, the ^Z was interpreted as 26 in decimal(\032). When ^Z is the first one of a line input, it is EOF(-1 in my system).

To sum up, windows thinks that, when the buffer is not empty(exclude ^Z), the input is not over yet. Only when the buffer is empty can that symbol be interpreted as an EOF.

Now, one question came to mind: how do we end an input at the end of a line instead of the beginning of a new one when we have to put a ENTER in front of the ^Z? The answer might be: using the ^Z to identify the end of the line input, which saves the program form reading in the following ENTER, while the ENTER key can still work as a buffer flusher. Next input ^Z at the beginning of the new line.

7. Prototyping for Checking

About argument passing validation in a function calling, when using the prototype without type specification, like int good(), different compiler gives different answers:

the following code, when compiling using gcc 8.1.0, nothing bad happened, no linkage error, however when compiling using the LLVM-clang 7.0.0, linkage error occurred.

quoting C Primer Plus Page352

Suppose you give a prototype like this:
void print_name();
An ANSI C compiler will assume that you have decided to forego function prototyping, and it
will not check arguments. To indicate that a function really has no arguments, use the void
keyword within the parentheses:
void print_name(void);
ANSI C interprets the preceding expression to mean that print_name() takes no arguments. It
then checks to see that you, in fact, do not use arguments when calling this function.
